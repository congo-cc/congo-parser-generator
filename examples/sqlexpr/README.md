# Building and Running the SQL Expression Parser

## Overview

The SQL Expression parser is a Rust parser for SQL boolean expressions
written in the JMS message-selector dialect.  The language is the SQL
fragment that appears inside a JMS message selector or a SQL `WHERE`
clause: boolean combinations of comparison expressions over typed
literals and variables.  It is designed to be used as a front end for
query filters, rule engines, message-routing rules, and similar
embedded expression languages — no database is required.

The grammar lives in [`SqlExprParser.ccc`](SqlExprParser.ccc) and is
processed by CongoCC's Rust target to produce an arena-based AST, a
recursive-descent parser, an NFA-based lexer, a visitor framework, and
a Wadler-Lindig pretty-printer.  Language features include:

- **Logical operators**: `AND`, `OR`, `NOT`
- **Equality**: `=`, `<>`, `!=`
- **Null tests**: `IS NULL`, `IS NOT NULL`
- **Comparison**: `>`, `>=`, `<`, `<=`
- **Pattern matching**: `LIKE`, `NOT LIKE`, with optional `ESCAPE` clause
- **Range**: `BETWEEN … AND …`, `NOT BETWEEN … AND …`
- **Set membership**: `IN (…)`, `NOT IN (…)` (with sign-prefixed numeric
  or string literals)
- **Arithmetic**: `+`, `-`, `*`, `/`, `%` and unary `+`/`-`
- **Literals**: decimal (with optional `L` suffix), hex (`0xFF`), octal
  (`0777`), floating-point (`1.5e-10`), strings (with `''` escapes),
  and the keywords `TRUE`, `FALSE`, `NULL`
- **Case-insensitive keywords**: `AND`, `and`, `aNd` all parse identically
- **Comments**: line (`-- …`) and block (`/* … */`)

The parser enforces only *syntax*.  Semantic constraints required by
JMS selectors (boolean-root enforcement, BETWEEN-bound type and order
checks, IN-list type uniformity) live in
[`rust-saved/tests/parser_test_support/mod.rs`](rust-saved/tests/parser_test_support/mod.rs)
— see [Test Code and Data](#test-code-and-data) below.

## Supported Syntax

```
-- Boolean operators
x > 5 AND y < 10
a = 1 OR b = 2
NOT active

-- Comparison operators
price >= 100
name <> 'admin'

-- Arithmetic in comparisons
(a + b) * c > 100

-- LIKE with wildcards (% and _) and optional ESCAPE
name LIKE 'John%'
code LIKE 'A_B%' ESCAPE '\'

-- BETWEEN (inclusive, bounds checked at parse time)
age BETWEEN 18 AND 65
temp BETWEEN -10.5 AND 100.0

-- IN lists (type-homogeneous)
status IN ('active', 'pending', 'completed')
code IN (100, 200, 300)

-- NULL checks
value IS NULL
value IS NOT NULL

-- Literals: integers, hex, octal, floats, scientific notation, long suffix
x = 42
flags = 0xFF
perms = 0755
rate = 3.14e-2
big = 1000000L

-- String literals with escaped quotes
name = 'It''s a test'

-- Identifiers with $ and _
$variable > 0
_internal = TRUE

-- Comments
x > 5 -- line comment
x /* block comment */ > 5
```

## Building the Rust Parser

### Generate the Rust Source Code

From the **examples/sqlexpr** directory, run the following command to build the rust SQL Expression parser:
```
java -jar ../congocc.jar -n -lang rust -d rust-sqlparser SqlExprParser.ccc
```

The same effect (plus copying testfiles and restoring the hand-written
tests in `rust-saved/`) can be obtained by running the ant target:
```
ant -Drust.enabled=true rust-gen
```

### Compiling the Parser

From the **rust-sqlparser** directory, issue *ONE* of the following commands for builds.  Replace "build" with "test" to build and then run all tests.  
See [README_RUST.md](https://github.com/richcar58/congo-parser-generator/blob/main/README_RUST.md#zero-dependencies) for details about activating the optional `serde` feature.  
```
cargo build                              # Debug build
cargo build --features serde             # Debug build with serde serialization support for AST types
cargo build --release                    # Optimized build
cargo build --release --features serde   # Optimized build with serde serialization support for AST types
```

`ant rust-gen` re-runs the parser generator and then copies
`testfiles/` to `rust-sqlparser/test-data/` and `rust-saved/` over
`rust-sqlparser/` with `overwrite="true"`, so any hand-written file is
preserved across regeneration.

### Test Data: `testfiles/`

The ten `.txt` files in `testfiles/` are short, representative SQL
expressions that exercise individual grammar features.  They are read
by the generated `tests/parse_files.rs` harness, which simply parses
each file and asserts that no parse error occurs.  Adding a new file
to `testfiles/` automatically enrolls it in the harness on the next
`ant rust-gen` (the build copies the entire directory into
`rust-sqlparser/test-data/`).

| File | Feature exercised |
|---|---|
| `arithmetic.txt` | `+ - * / %`, parentheses, comparison composition |
| `basic.txt` | Simplest equality predicate |
| `between.txt` | `BETWEEN … AND …` and `NOT BETWEEN …` |
| `comparison.txt` | `>=`, equality, AND-chaining |
| `complex.txt` | Line + block comments, nested OR/AND, LIKE wildcards |
| `in_list.txt` | `IN (...)` and `NOT IN (...)` over strings |
| `like_escape.txt` | `LIKE … ESCAPE '!'` and `NOT LIKE` |
| `literals.txt` | Boolean, hex, octal, scientific-notation literals |
| `negation.txt` | `NOT (...)`, unary minus, parenthesized predicates |
| `null_checks.txt` | `IS NULL` and `IS NOT NULL` |

### Test Programs: `rust-saved/tests/`

Five hand-written test files live in `rust-saved/tests/` and are
restored over the generated `rust-sqlparser/tests/` tree by
`ant rust-gen`.  Together they run **490 tests** against the generated
crate's public API.  Test names and intent are 1:1 ports from the
sibling project [`sqlexpr-congo-rust`](https://github.com/richcar58/sqlexpr-congo-rust),
so failures can be cross-referenced.

| File | # tests | What it does |
|---|---|---|
| `lexer_test.rs` | 62 | Exercises `parser::tokens` (`TokenType::is_regular/is_skipped/is_unparsed/is_more`, `TokenType::COUNT`, `Display`, `is_contextual_token`, `LexicalState::DEFAULT`) and `parser::lexer` (`Token` fields and text, `Lexer::new/tokenize/tokens/source_name`, token-set activation, lexical-state switching).  Positive cases verify documented behavior; negative cases cover invalid characters, unterminated string literals, empty input, and repeated invocations. |
| `parser_test.rs` | 164 | AST-shape tests: literals (decimal, hex, octal, float, boolean, NULL, string), every operator and operator combination, precedence and associativity, parenthesized expressions, complex multi-clause expressions, and parse-error inputs.  Uses `parser_test_support` helpers to assert on operator sequences and operand counts in the flat CongoCC AST. |
| `parser_test2.rs` | 155 | Broader feature smoke tests: boolean operators, relational operators, LIKE/BETWEEN/IN, `IS NULL` / `IS NOT NULL`, line and block comments, whitespace handling, and dozens of realistic selector expressions (price filters, status filters, audit filters, etc.).  Uses the simpler `parse(input) -> Result<(), ParseError>` wrapper from `parser_test_support`. |
| `parser_type_checking_tests.rs` | 97 | Semantic checks for `BETWEEN` bounds (literal-only, numeric type matching, ordering, NULL/Boolean rejection) and `IN` elements (literal-only, type-homogeneous, NULL/Boolean rejection).  These are enforced by `parser_test_support`'s validators, not by the generated parser. |
| `pretty_parse_test.rs` | 12 | Wadler-Lindig pretty-printer snapshot regression.  Each test parses a SQL fragment, runs `DefaultPrettyPrinter`, and asserts byte-for-byte that the rendered output matches the expected snapshot.  Collectively the suite exercises every grammar production and every literal kind at least once. |

In addition, `rust-sqlparser/tests/parse_files.rs` is **generated** and
not hand-written.  It runs once and processes every `.txt` file in
`test-data/`.

### `parser_test_support` — Shared Helpers and Semantic Validators

`rust-saved/tests/parser_test_support/mod.rs` is a shared support
module used by `parser_test.rs`, `parser_test2.rs`, and
`parser_type_checking_tests.rs`.  Cargo treats subdirectories under
`tests/` as common modules rather than as standalone integration
tests, so each consumer brings the module in with `mod
parser_test_support;`.

**Why it exists.**  The test files originated in the sibling
`sqlexpr-congo-rust` project, whose hand-written parser produces a
**typed** AST (e.g. `AstNode::EqualityExpression { operators: Vec<EqualityOp>, … }`)
and performs JMS-selector **semantic validation** (boolean root,
BETWEEN-bound types, IN-element types) inline as part of parsing.
The CongoCC-generated parser in this project instead produces a flat
`Ast` keyed by `NodeKind`, with operator tokens interleaved among
operand children, and performs only **syntactic** parsing — the Java
semantic-action INJECT block in the source grammar that enforces
boolean-root is currently not translatable to Rust by the generator.

To let the migrated tests express the same intent without per-test
rewriting, `parser_test_support`:

1. **Mirrors the source's operator enums** — `EqualityOp`,
   `ComparisonOp`, `AddOp`, `MultExprOp`, `UnaryOp` — so test bodies
   that assert on operator sequences port verbatim.
2. **Provides operator-extraction helpers** — `eq_ops`, `cmp_ops`,
   `add_ops`, `mult_ops`, `unary_op` — that read the interleaved
   operator tokens out of a node and return the same enum sequence
   that source tests assert against.
3. **Provides AST navigation helpers** — `operand_children`,
   `all_children`, `skip` (walks through single-child wrappers),
   `leaf_image`, `primary_value`, `is_token` — that make assertions
   like `eq.children.len() == 2` translate one-to-one.
4. **Re-implements the source's semantic validators in pure Rust** —
   `validate_boolean_root`, `validate_between_bounds`, and IN-element
   validation.  The top-level `parse_ok` / `parse_err` / `parse`
   entry points run these after the generated parser succeeds, so
   the source's `"parse error contains 'boolean'"` /
   `"BETWEEN bounds must be literal values"` / `"same type"`
   assertions still hold.

The three entry points exposed to test code:

| Function | Returns | Semantics |
|---|---|---|
| `parse_ok(input)` | `(Ast, NodeId)` | Parse must succeed and semantic validation must pass; pretty-prints the AST (visible with `--nocapture`); panics with a diagnostic otherwise. |
| `parse_err(input)` | `String` (the error message) | Either parsing must fail or semantic validation must fail; panics if input parses cleanly. |
| `parse(input)` | `Result<(), ParseError>` | Convenience wrapper for tests that just need a boolean ok/err result.  Pretty-prints the AST on success. |

## Running Tests

The Ant `test-rust` target builds the parser and runs the whole suite.
From the repository root:

```bash
ant -Drust.enabled=true -f examples/sqlexpr/build.xml test-rust
```

or from this directory:

```bash
ant -Drust.enabled=true test-rust
```

For finer-grained runs, work directly with `cargo` in
`rust-sqlparser/` after at least one `ant rust-gen`:

```bash
cd rust-sqlparser

# Whole suite, summary output only
cargo test

# Whole suite, with AST dumps and SQL inputs visible
cargo test -- --nocapture

# Just one test file
cargo test --test lexer_test
cargo test --test parser_test
cargo test --test parser_test2
cargo test --test parser_type_checking_tests
cargo test --test pretty_parse_test
cargo test --test parse_files

# Just one test (substring match against test names)
cargo test test_between_negative_numbers
cargo test test_between_negative_numbers -- --nocapture

# All tests whose names start with a prefix
cargo test test_like_      # all LIKE-related tests
cargo test test_in_        # all IN-list tests
```

`cargo test --test <name>` runs the integration test at
`tests/<name>.rs`.  The five hand-written files plus the generated
`parse_files` are addressable that way.  `cargo test <substring>`
filters by test function name across all test binaries.  Use
`-- --nocapture` to see the SQL input, the pretty-printed AST, and
the error messages that the helpers emit on each assertion.

## Editing Workflow

If you modify a test file in `rust-sqlparser/tests/`, copy the change
back into `rust-saved/tests/` (or `rust-saved/README.md` for the
crate-level README).  Otherwise the next `ant rust-gen` will overwrite
your edits with the older version from `rust-saved/`.
