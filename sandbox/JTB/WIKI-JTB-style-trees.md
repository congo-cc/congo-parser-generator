# Using CongoCC to Generate JTB-style Trees

For JavaCC, there are two distinct ways to generate a tree-structured representation of the parsed input text: the preprocessor-based **jjTree** tool and the **Java Tree Builder** (JTB) tool. While **CongoCC** includes built-in capabilities to produce a tree structure compatible with those generated by jjTree, it has, until now, lacked support for applications that rely on JTB-generated tree structures. The good news is—that is no longer the case.

A new capability in CongoCC allows its built-in tree builder to additionally generate tree nodes corresponding to those produced by the JTB tool. Moreover, the powerful reflection-based visitation supported by CongoCC’s tree nodes eliminates the need for JTB-generated visitor classes. Only the resulting CongoCC tree nodes are required to achieve JTB-style visitation with complete fidelity.

If this piques your interest, here’s how to enable it.

---

## Converting Your JavaCC/JTB Grammar to CongoCC

The first step is to render your JavaCC grammar file in a form acceptable to CongoCC. At one time, CongoCC maintained source compatibility with JavaCC. However, it became clear that preserving this compatibility detracted from the clarity of CongoCC grammars, particularly when using CongoCC’s extended features.

To address this, the source-compatible version of CongoCC (known as **JavaCC 21**) was frozen, and a rewriting utility was created. This tool converts most JavaCC grammars—including those preprocessed by JTB—into equivalent CongoCC grammars with minimal manual changes.

You can find information on using this conversion tool [here](https://github.com/congo-cc/congo-parser-generator/wiki/Migrating-to-CongoCC-%28from-JavaCC-21%29).

---

## Enabling JTB Tree Node Generation in Your CongoCC Grammar

To produce a tree structure compatible with JTB, you need to set a few CongoCC options at the beginning of your parser source file:

```
SMART_NODE_CREATION=false;
X_SYNTHETIC_NODES_ENABLED;
X_JTB_PARSE_TREE;
```

- `SMART_NODE_CREATION=false;` disables the default behavior of suppressing node creation when a node would otherwise contain no children or only a single child. JTB-style visitors expect all nodes implied by the grammar to be present.

- `X_SYNTHETIC_NODES_ENABLED;` enables an experimental feature that causes each syntactic expression or subexpression to potentially generate a **synthetic** node.

- `X_JTB_PARSE_TREE;` activates the generation of syntactic nodes at the same positions as JTB’s additional nodes and enables field definitions in non-terminal (production) nodes to reflect the top-level syntactic structure.

With these options enabled, the resulting tree structure will be nearly JTB-compatible. The final step is to define the synthetic node types and inject methods that JTB-generated visitors expect.

---

## Including JTB-Compatible Productions for CongoCC Synthetic Nodes

CongoCC provides two important features not found in JavaCC that make this process straightforward:

1. **INJECT**, which allows custom code to be injected into generated nodes.
2. **INCLUDE**, which lets you modularize the injected definitions.

To ensure compatibility, CongoCC’s synthetic nodes must extend the equivalent JTB classes. You can accomplish this with the following code:

```java
Terminal   : FAIL;
Sequence   : FAIL;
Choice     : FAIL;
ZeroOrOne  : FAIL;
ZeroOrMore : FAIL;
OneOrMore  : FAIL;

INJECT Terminal   : extends NodeToken
INJECT Sequence   : extends NodeSequence
INJECT Choice     : extends NodeChoice
INJECT ZeroOrOne  : extends NodeOptional
INJECT ZeroOrMore : extends NodeListOptional
INJECT OneOrMore  : extends NodeList
```

These injections rename the internal CongoCC nodes to match JTB’s expectations and ensure compatibility with JTB visitor methods.

Next, you define the required methods using the `INJECT` directive for each node class. This includes `NodeToken`, `NodeSequence`, `NodeChoice`, `NodeList`, `NodeListOptional`, and `NodeOptional`.



We add the methods as such:

```java
INJECT NodeToken :
    import java.util.*;
{     
    Token token = Token.newToken(INVALID, null, 0, 0);

    @Override
    public void close() {
        List<Token> tokens = (List<Token>) getAllTokens(false);
        if (tokens.size() > 0) {
            token = tokens.get(0);
        }
    }
    
    public Token getToken() {
        return token;
    }
    
    @Override
    public String toString() {
        return token.toString();
    }
    
    @Deprecated
    @Override
    public String getImage() {
        return token.toString();
    }
    
    @Override
    public String toCompleteString() {
        return token.toCompleteString();
    }

    @Override
    public int getLine() {
        return token.getLine();
    }

    @Override
    public int getLastLine() {
        return token.getLastLine();
    }

    @Override
    public int getColumn() {
        return token.getColumn();
    }

    @Override
    public int getLastColumn() {
        return token.getLastColumn();
    }
    
    @Override
    public boolean isEOF() {
        return token.getType() == Token.TokenType.EOF;
    }

    public NodeToken() {
        super();
    }

    @Override
    public int hashCode() {
        return Objects.hash(
                   token.getBeginOffset(), 
                   token.getEndOffset(), 
                   token.toString(), 
                   token.getParent(), 
                   token.getSource(), 
                   token.getTokenSource(), 
                   token.getType()
                   );
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null) return false;
        if (getClass() != obj.getClass()) return false;
        NodeToken other = (NodeToken) obj;
        return getBeginOffset() == other.getBeginOffset() && 
               getEndOffset() == other.getEndOffset() && 
               Objects.equals(toString(), other.toString()) && 
               Objects.equals(getParent(), other.getParent()) && 
               Objects.equals(getSource(), other.getSource()) && 
               Objects.equals(getTokenSource(), other.getTokenSource()) && 
               getType() == other.getType() && 
               isUnparsed() == other.isUnparsed();
    }
}
```

These definitions equip the synthetic nodes with the methods necessary for full JTB visitor compatibility.

## Modifying Visitors to Work with CongoCC's JTB-Compatible Node Trees

To use your existing JTB-style visitors with the new CongoCC-generated node trees, make the following minor changes:

1. **Update the base class**:

   Change your class definition from:

   ```java
   public class MyJtbVisitor extends DepthFirstVoidVisitor {
   ```

   to:

   ```java
   public class MyJtbVisitor extends Node.Visitor {
   ```

2. **Remove `@Override` annotations**:

   Your visitor methods may look like this:

   ```java
   @Override
   public void visit(ANonTerminal n) {
       ...
   }
   ```

   Remove the `@Override` annotation, since you are not overriding a specific method signature from the CongoCC visitor class.

3. **Update `super.visit(n);` calls**:

   If your methods contain `super.visit(n);`, replace them with:

   ```java
   recurse(n);
   ```

4. **Update or define `accept` behavior**:

   If your code uses `n.accept(this);`, you can either:

   - Replace `n.accept(this);` with `visit(n);`, **or**
   - Inject the `accept(...)` method into the base `Node` type by adding:

     ```java
     INJECT Node :
     {
         /**
          * Accepts a visitor to this {@code Node}.
          * @param visitor is a {@link Node.Visitor} to this node
          */
         default public void accept(Visitor visitor) {
             visitor.visit(this);
         }
     }
     ```

     It’s recommended to place this in a separate file and use the `INCLUDE` directive to add it to your parser source.

## That’s All, Folks!

You now have a CongoCC grammar that can be compiled into Java code that produces a parse tree compatible with JTB visitors—without requiring changes to the visitor logic beyond the small adjustments described above.

This new capability allows you to migrate your JavaCC/JTB-based projects to CongoCC while retaining your existing tree traversal infrastructure.

## Note:

If any tree node annotations (i.e., denoted by "#") are used in a production, JTB-style synthetic nodes and fields will be suppressed for the associated node and all of its descendants in the same production.  If the annotation is placed on the production node itself, no JTB synthetic nodes or fields will be generated for that production.  This gives you the ability to utilize advanced CongoCC features to incrementally convert your pre-existing JTB nodes and visitors to pure-CongoCC if desired.
